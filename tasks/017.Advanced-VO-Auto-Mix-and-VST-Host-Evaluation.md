# Task 017 — Advanced VO Auto-Mix & VST Host Evaluation

Objective: Improve narration-over-music intelligibility with a more production-grade auto-mix chain, while explicitly evaluating whether a dedicated VST host is necessary for offline rendering.

Background
- Current renderer path already supports deterministic music ducking via ffmpeg filtergraphs in `services/renderer/compiler/short.py`.
- The immediate problem space is mix quality and consistency, not browser playback or real-time monitoring.
- A VST host adds substantial operational complexity, so it should be gated behind measurable shortcomings in the ffmpeg-based pipeline.

Implementation Phases
- [ ] Phase 1: upgrade the ffmpeg auto-mix chain for offline rendering.
- [ ] Phase 2: define objective and subjective acceptance metrics for voice clarity, music loudness, and clipping/headroom.
- [ ] Phase 3: only if Phase 1 fails acceptance, prototype a server-side VST host path for offline batch rendering.

Phase 1 — ffmpeg Auto-Mix Improvements
- [ ] Upmix narration explicitly to stereo before mixing.
- [ ] Add narration conditioning: high-pass filter, light compression, and loudness normalization.
- [ ] Add music conditioning: stereo normalization plus optional EQ cuts in the speech-presence range.
- [ ] Tune sidechain ducking with explicit attack, release, ratio, threshold, and makeup controls.
- [ ] Add final bus protection with limiter/true-peak guard so renders do not clip.
- [ ] Expose operator-safe env or preset knobs for the mix chain without requiring code edits.

Phase 2 — Validation & Tooling
- [ ] Add ffprobe- and filter-based checks for channel count, peak level, and integrated loudness where practical.
- [ ] Add fixture renders covering: VO only, VO + quiet music, VO + dense music, and long pauses in narration.
- [ ] Add manual review notes to the runbook for evaluating intelligibility, pumping, and music recovery between phrases.
- [ ] Record representative before/after samples for regression comparison.

Phase 3 — VST Host Evaluation Gate
- [ ] Document concrete reasons the ffmpeg chain is insufficient if Phase 1 fails.
- [ ] Identify candidate plugin-hosting approaches suitable for headless offline rendering on the renderer worker.
- [ ] Evaluate determinism, licensing, containerization, CPU cost, preset portability, and crash isolation.
- [ ] Prototype a minimal offline chain with sidechained ducking and one mastering stage.
- [ ] Keep the VST path optional and server-only; no plugin or admin controls may leak to the browser.

Tests
- [ ] Unit: compiled render commands include the configured auto-mix stages and stereo output guarantees.
- [ ] Integration: rendered output remains stereo, narration stays centered, and music audibly ducks during speech.
- [ ] Integration: loud sections do not clip and quiet sections recover naturally after narration gaps.
- [ ] Failure: invalid mix configuration produces actionable renderer logs and marks the job `errored`.

Acceptance Criteria
- [ ] Default renders produce centered narration with consistent intelligibility over background music.
- [ ] Output remains deterministic for the same inputs and preset values.
- [ ] Renderer operators can tune the mix without changing code.
- [ ] A VST host is introduced only if the ffmpeg path demonstrably cannot meet the quality bar.
